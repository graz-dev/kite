# Internal CRD Reference

This document describes the CRD types from an implementation perspective.
For end-user documentation see the [public CRD reference](../public/docs/crd-reference.md).

## OptimizationTarget

### Key behaviours

- **Cluster-scoped**: no namespace in the object name.
- **Finalizer**: `optimization.kite.dev/finalizer` is added on first reconcile
  and removed on deletion.  The finalizer currently only performs in-cluster
  cleanup (removing the finalizer itself); external resources (PRs, branches)
  are not cleaned up.
- **Status subresource**: status updates use `r.Status().Update()` to avoid
  conflicts with spec updates.

### Condition types

| Type | True condition | False condition |
|------|----------------|-----------------|
| `Ready` | Normal reconcile | Configuration error or analysis failure |

### Fields with complex defaults

`RightsizingRules` has zero-value defaults applied in `algorithm.defaultRules()`:

| Field | Zero value | Applied default |
|-------|------------|-----------------|
| `CPUPercentile` | 0 | 95 |
| `MemoryPercentile` | 0 | 100 |
| `CPUSafetyMarginPercent` | 0 | 15 |
| `MemorySafetyMarginPercent` | 0 | 15 |
| `MinCPURequest` | zero Quantity | `10m` |
| `MinMemoryRequest` | zero Quantity | `32Mi` |
| `HPABehavior` | empty string | `Include` |

This allows the CRD to omit these fields and still get sensible behaviour
without requiring `+kubebuilder:default` markers that are difficult to maintain
for complex types.

## MetricsHistory

### Naming

Names are generated by `metrics.historyName(namespace, name, kind)`:

```go
raw := strings.ToLower(fmt.Sprintf("%s-%s-%s", kind, namespace, name))
// e.g. "deployment-production-my-api"

if len(raw) <= 63:
    use raw (sanitised)
else:
    short := raw[:50] + hex(sha256(raw)[:4])
    // e.g. "deployment-production-my-very-long-workload-na-a1b2c3d4"
```

Sanitisation replaces any character that is not `[a-z0-9-]` with `-` and
trims leading/trailing dashes.

### Data pruning

On every `upsertMetricsHistory` call:
1. Append the new data point.
2. Filter out points older than `historyWindow`.
3. If the remaining count exceeds `maxDataPoints`, keep the newest N points.

This ensures the CRD objects do not grow unboundedly.

### Storage estimate

Default settings: 5-minute scrape, 7-day window = 2016 points per workload.
Each data point contains:
- `timestamp` (30 bytes)
- `replicaCount` (4 bytes)
- Per container: `name` (~15 bytes) + `cpuMilliCores` (8 bytes) + `memoryBytes` (8 bytes)

For a workload with 3 containers: ~100 bytes/point × 2016 = ~200 KB per workload.

At 100 workloads: ~20 MB of etcd data, well within typical etcd limits.

## RBAC requirements

The operator requires the following permissions:

| Resource | Verbs | Reason |
|----------|-------|--------|
| `optimizationtargets` | CRUD | Main CRD management |
| `optimizationtargets/status` | get, update, patch | Status subresource |
| `optimizationtargets/finalizers` | update | Finalizer management |
| `metricshistories` | CRUD | History storage |
| `deployments, statefulsets, daemonsets` | get, list, watch | Workload discovery |
| `horizontalpodautoscalers` | get, list, watch | HPA detection |
| `pods` | get, list, watch | Pod discovery for metrics |
| `namespaces` | get, list, watch | Namespace auto-discovery |
| `secrets` | get | GitOps token reading |
| `events` | create, patch | Kubernetes event recording |
| `metrics.k8s.io/pods` | get, list | metrics-server queries |
| `leases` | CRUD | Leader election |
